# Part 1 Plan for `r/fitness` with Part 2-Compatible Data Design

## Summary
- Build a Python-first local pipeline and Streamlit application for `r/fitness`, using Arctic Shift instead of PRAW, SQLite as the required local database, and precomputed NLP outputs over posts plus comments. Justification: this matches your Python preference, satisfies the assignment, and avoids a later stack rewrite.
- Automatically select a contiguous Arctic Shift time window spanning at least 6 months and containing at least 15,000 posts, then ingest all comments for those posts. Justification: this guarantees rubric compliance without guessing dates manually.
- Preserve raw payloads, cleaned analysis text, unified document records, and an embedding-ready schema so Part 2 can add retrieval without re-ingesting or redesigning the corpus. Justification: this keeps Part 1 in scope while preventing an awkward migration later.

## Key Interfaces
- Offline CLI stages: `ingest_posts`, `ingest_comments`, `prepare_features`, `fit_topics`, `fit_stance`, `build_app_cache`. Justification: heavy work should happen before app launch so the frontend stays responsive and reproducible.
- SQLite tables: `posts`, `comments`, `authors`, `subreddit_meta`, `documents`, `document_embeddings`, `topic_definitions`, `document_topics`, `topic_time_series`, `comment_stances`, `topic_argument_summaries`. Justification: this is the smallest schema that supports Part 1 cleanly and leaves a clear hook for Part 2 embeddings.
- App pages: `Overview`, `Topics`, `Topic Detail`, `Methods`. Justification: these pages map directly to the rubric and make your design choices visible to the grader.

## Implementation Changes
- Acquisition: use an Arctic Shift aggregate pass to identify a valid `r/fitness` date window, then fetch posts in ascending date order and store both raw JSON and normalized fields. Justification: the aggregate pass reduces guesswork, and ordered ingestion simplifies retries.
- Rate limiting and reruns: implement paginated ingestion with checkpointing, backoff/retry handling, and idempotent upserts keyed by Reddit IDs for both posts and comments. Justification: Arctic Shift may fail mid-run, so partial reruns must not double-count or corrupt relations.
- Comment ingestion: fetch all comments for selected posts, preserve `link_id` and `parent_id`, and keep deleted/removed comments in storage while excluding them from modeling. Justification: stance analysis and later RAG need thread structure, but corpus statistics should still reflect the true archive.
- Cleaning: keep `raw_text` and `clean_text` side by side; remove markdown noise, URLs, and boilerplate for modeling; exclude AutoModerator and likely bots from NLP analysis only. Justification: this improves model quality without distorting database-level counts.
- Canonical document layer: create a unified `documents` table for posts and comments with stable `document_id`, `source_type`, `created_utc`, `author_id`, `parent_id`, `link_id`, `clean_text`, and SQLite FTS support. Justification: this supports Part 1 exploration and makes Part 2 retrieval setup straightforward.
- Embedding forward-compatibility: add a `document_embeddings` table keyed by `document_id` with nullable storage for vector bytes/model metadata, but do not populate it in Part 1. Justification: this avoids a schema migration in Part 2 without pulling RAG implementation into this phase.
- Author handling: store raw usernames internally but display stable pseudonymous aliases in the app. Justification: user grouping is required for the rubric, while pseudonyms are easier to defend in the ethics discussion.
- Topic modeling corpus: fit topics on a hybrid document corpus built from all model-ready posts plus a bounded sample of model-ready top-level comments. The comment sample should rank comments by score within each post, keep the top 3 per post, and then apply a global cap to comments only so the final modeling corpus remains around 15,000 documents while never squeezing out posts. Justification: `r/fitness` is megathread-driven, the TA requested that posts still factor into topic modeling, and this hybrid rule preserves post representation while letting comments carry the real conversational signal.
- Topic modeling: use MiniLM embeddings with BERTopic-style clustering and outlier reduction over the hybrid document corpus, target 8 human-facing topics, and tune/reduce so the final displayed topic count remains within the required 5-20 range. Justification: 8 is a practical target for readability, but the actual deliverable must respect the rubric rather than a hardcoded count.
- Topic label review: for each final topic, review the top 10 keywords and rename the autogenerated label only if it is misleading or too vague. Justification: this is a defensible light review pass that improves usability without turning the task into manual annotation.
- Topic outputs: each topic must show a short label, 5-10 keywords, share of modeled documents, and representative documents, with the UI able to distinguish post versus comment exemplars. Justification: the hybrid corpus changes the modeling unit from posts alone to unified discussion documents while still covering item 1.2 cleanly.
- Trending vs persistent: compute monthly topic share over the selected window using the hybrid topic-model corpus; label `Persistent` when a topic appears in at least 75% of months with low volatility, `Trending` when activity is concentrated in a sharp 1-2 month spike, and `Mixed` otherwise. Justification: these rules remain objective and explainable, and they match the actual modeling unit.
- Stance scope: run stance analysis for every topic designated as a `major topic`, where `major` is defined in the Methods page as any displayed topic whose post share clears a fixed minimum threshold; if this still produces too many low-signal topics, keep all major topics but explicitly note the thresholding decision in the report. Justification: this satisfies item 1.4 while making the definition intentional and defensible.
- Stance method: cluster substantive topic-specific comments into two candidate camps using MiniLM embeddings, treat the larger upvote-weighted cluster as the provisional dominant position, and classify eligible comments by nearest prototype. Justification: this gives a fully local unsupervised stance pipeline that is realistic on a laptop.
- Consensus guardrail: only surface two-sided stance summaries when the minority cluster exceeds a minimum support threshold and produces coherent representative comments; otherwise mark the topic as `Mostly agreement / weak opposition signal` and suppress a strong opposing summary. Justification: this avoids presenting noise as meaningful disagreement.
- Methods disclosure: explicitly document that unsupervised stance detection can over-separate near-consensus topics and may miss subtle rhetorical differences. Justification: this addresses a real methodological limitation before the grader has to infer it.
- Argument summaries: generate and cache extractive summaries for each valid stance at pipeline time in `topic_argument_summaries`, then read them in the app rather than summarizing on demand. Justification: cached summaries are faster, deterministic, and easier to inspect/debug.
- UI behavior: `Overview` shows corpus stats and timeline activity; `Topics` shows topic cards and trend tags; `Topic Detail` shows representative documents, monthly trend, stance split, grouped users, and side-by-side argument summaries; `Methods` explains every design choice and its justification, including why `r/fitness` topics are modeled on a hybrid post+comment corpus. Justification: the app itself should demonstrate both the outputs and the reasoning behind them.

## Test Plan
- Verify the selected window spans at least 6 months and contains at least 15,000 posts. Justification: these are the non-negotiable assignment constraints.
- Verify ingestion reruns are idempotent after an interrupted job and do not double-insert posts/comments under Arctic Shift retries or partial failures. Justification: rate limits and mid-run interruptions are likely operational risks.
- Verify all comments maintain valid `link_id` and `parent_id` references and that `documents` reconciles exactly with `posts + comments`. Justification: relational correctness is necessary for both stance analysis and future retrieval.
- Verify deleted/removed content is counted in corpus statistics but excluded from topic/stance modeling. Justification: this keeps counts faithful while protecting model quality.
- Verify the final displayed topic count falls within 5-20, with 8 as the target rather than a forced constant, and verify that all model-ready posts are included in the topic corpus before the comment cap is applied. Justification: this closes the main rubric gap around topic count while preserving the TA-requested post representation.
- Verify each major topic receives a trend label and stance result or an explicit `weak opposition signal` message. Justification: this ensures item 1.4 is handled intentionally, not silently skipped.
- Manually audit 20-30 stance-labeled comments across topics for directional sanity and inspect minority-cluster summaries for noise. Justification: stance is the highest-risk analytical component.
- Verify cached summaries in `topic_argument_summaries` match the topic/stance pairs shown in the app and that the app loads without recomputing NLP artifacts. Justification: this protects demo stability.

## Assumptions
- Use Streamlit, Python 3.11+, SQLite, Plotly, pandas, and sentence-transformers in a free, CPU-first workflow. Justification: this is the most practical Python-heavy path for you.
- Do not implement Part 2 now, but preserve document-level IDs, metadata, FTS support, and embedding-ready storage so retrieval can be added without schema redesign. Justification: this keeps scope tight while honoring the forward-compatibility requirement.
- Use one light manual review step only for topic label cleanup based on top keywords and representative posts; all other outputs remain pipeline-generated. Justification: this is transparent, limited, and easy to justify in the report.
- Treat `r/fitness` as a comment-heavy, megathread-driven community where posts must still be represented in topic modeling, but comments carry most of the semantic signal. Justification: this matches the observed corpus structure and the TA guidance without forcing a weak post-only model.
